## 1. 工程原则

- Agent 只处理推理、创意、判断和改写；确定性逻辑做成 Service。
- Agent 输出必须用 Pydantic schema 校验后再入库。
- 能用 LangGraph 原生能力、官方扩展、成熟插件或稳定第三方库解决的流程编排、checkpoint、interrupt/resume、重试、状态持久化、队列、存储和解析问题，不自造轮子；确需自研时，必须说明现有方案为何不满足当前业务边界。
- Provider、视频规格、模型和成本规则必须来自配置或抽象接口，不能写死在业务代码里。
- 真实视频生成、真实 API 调用或任何会增加真实成本的动作，都必须可见、可估算、可追踪，并在既定人工确认节点之后执行。
- 需要跨请求、跨进程、刷新、重启、审计、重跑或前端展示的数据，必须进入持久化存储或使用持久化组件；内存方案只能用于明确无需保留的数据，并在代码或文档中写明原因。
- 遵守单一来源原则：同义字段、枚举、状态和业务规则只维护一份；能由既有状态推导的内容，不新增重复字段。

## 2. 代码和字段规范

- 后端模型字段、Schema 字段、配置字段和 `GraphState` 字段必须有中文说明。
- 注释解释业务含义、边界和关键判断原因，不写“给变量赋值”这类无意义注释。
- 涉及 `GraphState`、数据库模型、迁移字段、工作流状态、节点运行记录、Provider 抽象或核心工作流的变更，必须先说明字段归属、是否入库、对刷新恢复、重跑和节点跳转的影响，并同步维护对应权威文档。
- 优先在既有稳定抽象上增量扩展；确需重命名、删除、迁移或替换已进入代码、数据库设计或接口约定的结构时，必须说明兼容/迁移方案并经用户确认。

## 3. 协作确认原则

遇到以下情况，必须先向用户提出问题并获得确认，再进行代码修改：

- 需求描述存在歧义。
- 用户最新要求与已确认的产品逻辑、工程原则、数据模型、成本策略或实现边界冲突。
- 资料、价格、模型能力、供应商限制、免费额度或 API 行为可能不准确或可能变化。
- 文档、代码和用户最新表述互相冲突。
- 改动会影响数据模型、数据库迁移、任务状态机、Provider 抽象或核心工作流。
- 改动会明显增加个人开发复杂度、运行成本或真实视频生成成本。
- 需要选择新的第三方供应商、SDK、队列方案、UI 库或存储方案。
- 会删除、覆盖、迁移或重构已有代码和数据。
- 无法确定某个实现属于第一版 MVP、后续增强还是应暂缓事项。

可以直接推进：

- 用户已经明确要求修改某个文档或代码点。
- 修改只是在既定方案内补充字段、文案、注释或低风险配置。
- 修复明确的拼写、链接、格式或小范围一致性问题。

即使可以直接推进，也要在动手前简要说明将修改什么。真实视频 API 调用、成本消耗、破坏性操作和大范围重构必须额外确认。

## 4. 逻辑冲突处理流程

发现用户逻辑冲突时，不能直接写代码，必须先：

1. 明确指出冲突点。
2. 说明继续执行会带来的产品、工程、成本或稳定性风险。
3. 给出至少一条推荐路线，并说明为什么更适合当前项目阶段。
4. 等用户确认选择后，再开始修改代码或文档。

推荐路线优先考虑：

- 个人开发可实现性。
- 成本可控。
- 工程稳定。
- 第一版 MVP 边界。
- 后续扩展空间。
