# 可行性、成本和商业边界

日常开发优先阅读 [Project Brief](00-project-brief.md)。本文件只记录可行性、成本、稳定性和商业判断；MVP 范围不在这里重复维护。

## 1. 结论

项目可行，但必须围绕 Project Brief 的 MVP 边界做。个人开发者适合先完成“可演示、可试单、可导出”的闭环，不适合第一版扩成通用 SaaS、复杂剪辑器、多 Agent 技术展示平台或投放系统。

第一版按可维护和可扩展优先，选择 LangGraph StateGraph 做受控 Agent 工作流，而不是自由聊天式多 Agent。用户为结果、速度、成本和稳定性付费，不会为 AutoGen、Microsoft Agent Framework 或 CrewAI 框架本身付费。

Review & Cost Gate 例外保留局部多 Agent，因为它直接服务市场验证：减少无效真实生成、提高商品保真判断、解释风险和成本，增强用户确认前的信任。

## 2. 市场判断

用户不缺“又一个视频生成器”，更有价值的切口是：

- 更快地产出多个商品素材方案。
- 围绕卖点、平台和风格做变体。
- 控制生成成本和失败重试。
- 保留审核记录和可复跑工程文件。
- 支持运营、剪辑师或商家继续加工。

## 3. 个人实现瓶颈

主要瓶颈不是 CRUD，而是真实视频生成的不确定性：

- 供应商能力、价格和免费额度会变化。
- 商品保真、文字和 Logo 生成质量不可完全控制。
- 轮询、下载、转存、重试和失败恢复容易拖慢开发。
- FFmpeg 合成可以工程化，但字幕精准对齐和视频局部修复会显著增加复杂度。

控制方式：

- LangGraph 负责固定图工作流和 human-in-the-loop，减少手写流程状态分散导致的维护成本。
- Pydantic schema 强化结构化输出，减少自由多 Agent 对话带来的不确定性。
- Prompt Review & Edit / Prompt Check 放在真实生成前，降低用户改坏 Prompt 后直接烧视频成本的风险。
- Review & Cost Gate 只在真实生成前运行，避免把多 Agent 成本扩散到全流程。
- SQLModel、SQLAdmin、TanStack Query 和 OpenAPI 类型生成用于减少样板代码，提高个人开发速度。
- 先完成 mock 闭环，再接真实供应商。
- 每个镜头独立生成、独立重试、独立记录成本。
- 真实生成前要求人工确认和预算校验。
- 不把 video-edit、精准字幕、自动投放放进第一版主路径。

## 4. 成本原则

以按秒计费的视频模型为基准，成本控制必须内置在产品流程里：

- 执行 Project Brief 的视频规格分档策略。
- 真实模型只用于连通性、镜头质量测试、演示和试单。
- 默认 720P，展示或交付时再考虑 1080P。
- 免费额度只用于连通性测试，不作为长期商业模型。
- 完整真实生成前必须展示预计成本。
- 单镜头和单项目必须有预算上限。

## 5. 稳定性判断

工程流程可以稳定，AI 视频结果不能保证稳定。

需要默认存在的问题：

- 商品外观不准确。
- 包装文字或 Logo 变形。
- 人物、手部、液体和复杂动作不自然。
- 同一 Prompt 多次结果差异明显。
- 视频生成排队、超时或失败。
- 供应商临时 URL 过期。

兜底策略：

- 每个镜头独立生成和重试。
- 真实生成前人工确认。
- 生成后立即下载并转存。
- 审核清单由用户确认。
- 即使视频不完美，也能导出脚本、分镜和 Prompt 作为交付物。

## 6. 商业验证建议

第一阶段不建议直接做纯 SaaS 等自然增长。更适合：

1. 先做可演示 MVP。
2. 找 10 个电商商家、运营或代运营测试。
3. 用服务包验证付费意愿。
4. 把重复交付动作沉淀为工具功能。

建议试单价格：

- 299-599 元 / 商品 / 3 条视频方案。
- 599-999 元 / 商品 / 5-10 条素材变体。

工具订阅后续可尝试：

- 99-199 元/月，用户自带 API Key。
- 如果平台承担真实生成成本，必须额外按条、按秒或 credits 计费。
